Class and Package Summary (Javadoc Overview)

All source files are part of the package:

    package MedicationTracking

Each class includes Javadoc-style comments to describe its purpose and methods.

Example:

/**
 * Represents a doctor in the pharmacy system.
 * Each doctor has a specialization and manages a list of patients.
 */
public class Doctor extends Person {
    private String specialization;
    private List<Patient> patients;

    public Doctor(String name, int age, String phoneNumber, String specialization) {
        super(name, age, phoneNumber);
        this.specialization = specialization;
        this.patients = new ArrayList<>();
    }

    public String getSpecialization() { return specialization; }
    public List<Patient> getPatients() { return patients; }

    public void setSpecialization(String specialization) { this.specialization = specialization; }

    public void addANewPatient(Patient patient) { this.patients.add(patient); }

    @Override
    public String toString() {
        return String.format("Doctor ID:%d Name:%s Specialization:%s Patients:%d", 
               getId(), getName(), specialization, patients.size());
    }
}

Build Process

1. Navigate to the source folder:

    cd JAVA-GROUP4-LEAH-STEPHEN

2. Compile all .java files:

    javac MedicationTracking/*.java

3. Run the program:

    java MedicationTracking.Menu

Development Standards

| Standard               | Description                                                        |
| ---------------------- | ------------------------------------------------------------------ |
| **Naming Conventions** | Classes use PascalCase, variables and methods use camelCase.       |
| **File Organization**  | Each class resides in its own `.java` file under the same package. |

Theoretical Database Design

While this project does not implement a database, the entities within the program have been structured so they can easily
map to relational tables in a real-world scenario.

Diagram:

[Doctor] 1 --- * [Prescription] * --- 1 [Patient]
                        |
                        *
                   [Medication]

Potential Table Structure

Doctor:

| Column         | Type     | Description       |
| -------------- | -------- | ----------------- |
| doctor_id      | INT (PK) | Unique doctor ID  |
| name           | VARCHAR  | Doctor’s name     |
| age            | INT      | Age               |
| phone          | VARCHAR  | Contact number    |
| specialization | VARCHAR  | Area of expertise |

Patient:

| Column     | Type     |
| ---------- | -------- |
| patient_id | INT (PK) |
| name       | VARCHAR  |
| age        | INT      |
| phone      | VARCHAR  |

Medication:

| Column        | Type     |
| ------------- | -------- |
| medication_id | INT (PK) |
| name          | VARCHAR  |
| dose          | VARCHAR  |
| quantity      | INT      |
| expiry_date   | DATE     |

Prescription:

| Column              | Type     | Description                    |
| ------------------- | -------- | ------------------------------ |
| prescription_id     | INT (PK) | Unique ID                      |
| doctor_id           | INT (FK) | Linked to `Doctor`             |
| patient_id          | INT (FK) | Linked to `Patient`            |
| medication_id       | INT (FK) | Linked to `Medication`         |
| prescription_expiry | DATE     | Defaults to 1 year after issue |

Entity Relationships

| Relationship              | Description                                                   |
| ------------------------- | ------------------------------------------------------------- |
| Doctor → Patient          | One doctor can manage multiple patients.                      |
| Patient → Prescription    | One patient can have multiple prescriptions.                  |
| Prescription → Medication | Each prescription references one medication.                  |
| Medication                | Independent table used in prescriptions and stock management. |
